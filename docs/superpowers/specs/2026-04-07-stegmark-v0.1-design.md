# StegMark v0.1 Design

> Scope baseline extracted from `docs/设计文档.md` and `docs/实施计划.md`.
> Approved implementation target for the first complete deliverable.

## Goal

Deliver a usable image invisible-watermark tool that can be installed with `pip`, run locally without model downloads, and expose both CLI and Python API for embed/extract/verify/info workflows.

## Scope

This iteration implements the `v0.1` milestone only.

Included:

- Python package scaffold with typed public API
- CLI commands: `embed`, `extract`, `verify`, `info`
- Default built-in watermark engine with no heavy ML dependency
- Optional external `trustmark` adapter when the upstream package is available
- Image I/O, metadata preservation, message codec, error model, config loading
- Unit tests and CLI tests
- Packaging metadata and user-facing README

Explicitly deferred:

- Self-trained HiDDeN and StegaStamp engines
- ONNX export and model download manager
- Benchmark suite
- REST API, Docker, MkDocs site

## Architecture

The package keeps the engine abstraction from the design document so later versions can add ML-backed engines without rewriting the CLI or API.

Layers:

1. `core`: engine abstraction, native engine, optional TrustMark adapter, codec, image I/O
2. `service`: high-level embed/extract/verify/info orchestration and result objects
3. `cli`: Click-based commands and formatting
4. `api`: simple top-level functions plus `StegMark` session object

## Engine Strategy

`auto` selects engines in this order:

1. External `trustmark` adapter if the dependency is installed and usable
2. Built-in native engine

The built-in engine is the required baseline and must always work offline. It uses a deterministic tiled frequency-domain watermark so the package remains lightweight and testable. The TrustMark adapter is optional integration, not a hard dependency.

## Watermark Format

- Input message supports text and explicit bit strings
- Messages are normalized to bytes, then encoded into a framed bitstream
- Frame format: version byte, payload length, payload bytes, CRC32
- The native engine stores repeated frame bits across image blocks and reconstructs by majority vote

This is simpler than the future BCH-based design but keeps the codec boundary stable for later replacement.

## User Experience

Default command shape stays aligned with the original docs:

- `stegmark embed input.png -m "Alice"`
- `stegmark extract input_wm.png`
- `stegmark verify input_wm.png -m "Alice"`
- `stegmark info input_wm.png`

JSON output is supported for automation. Human-readable output is the default.

## Error Handling

Errors remain explicit and typed:

- invalid input and unsupported formats
- image too small
- message too long for the selected engine
- decode failure / watermark not found
- unavailable engine dependency

Every public exception exposes a short `hint`.

## Testing

The first deliverable must prove:

- embed then extract round-trip on generated fixtures
- verify returns correct boolean/exit code
- CLI commands operate on files and JSON output
- invalid input surfaces typed exceptions
- metadata-preserving save/load path works for common formats

## Success Criteria

- `pip install -e .` works
- CLI commands are usable on generated images
- Python API round-trips messages
- test suite passes locally
- package layout is ready for future `hidden` and `stegastamp` engines
