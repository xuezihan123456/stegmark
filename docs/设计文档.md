## 1. 项目定位与战略

### 1.1 一句话定位

> **StegMark — 一条命令给图片加隐形水印。`pip install stegmark`，轻量、鲁棒、开源。**

### 1.2 核心目标

- **开源工具导向**：追求 GitHub stars 和社区影响力
- **极致易用**：`pip install stegmark` + 一条命令即可用
- **真正轻量**：推理仅依赖 ONNX Runtime (~15MB)，不强制依赖 PyTorch (~700MB+)
- **质量驱动**：无硬性 deadline，做好一个版本发一个

### 1.3 目标用户（按优先级）

| 优先级 | 用户群体 | 核心需求 | 触达渠道 |
|--------|---------|---------|---------|
| P0 | 内容创作者/摄影师 | 一条命令保护作品，不影响画质 | 少数派、即刻、摄影社区 |
| P1 | 开发者/工程师 | Python API 集成，批量处理 | GitHub、Reddit、HN |
| P2 | AI 安全从业者 | AI 生成内容溯源标记 | Twitter/X ML 社区 |
| P3 | 学术研究者 | 可复现 baseline，可引用 | 知乎学术圈、arXiv |

### 1.4 战略选择：混合策略（瑞士军刀 + 自研引擎）

经过深入竞品分析，我们采用**混合策略**而非纯自研路线：

```
                  StegMark (统一 CLI + Python API)
                 /           |           \
          TrustMark      自研 HiDDeN    自研 StegaStamp
          (Adobe预训练)   (v0.2+)       (v0.4+)
          v0.1 快速上线   学习+迭代      长期差异化
```

**理由**：
1. **v0.1 速度**：包装 TrustMark ONNX 模型，一周内可发布 MVP，抢占社区心智
2. **学习价值**：自研 HiDDeN/StegaStamp 对深度学习能力是极大锻炼，是简历加分项
3. **长期差异化**：自研模型可针对中文场景、特定攻击优化，不受上游限制
4. **用户透明**：同样的 `stegmark embed` 命令，底层引擎可切换

### 1.5 水印规格

| 参数 | 值 | 说明 |
|------|-----|------|
| 信息类型 | 短文本 + UUID/哈希 | 统一为二进制比特串 |
| 有效载荷 | 32-100 bits | 默认 48 bits（BCH 编码后 63 bits） |
| 图片格式 | PNG + JPEG + WebP | 输入输出均支持 |
| 最小尺寸 | 128×128 | 更小的图片拒绝处理 |
| 尺寸适配 | 自动适配任意尺寸 | 内部缩放处理，残差上采样回原尺寸 |

## 2. 竞品分析与差异化

### 2.1 竞品格局（2026 年）

| 竞品 | 维护方 | License | Stars | 状态 | 优势 | 劣势 |
|------|--------|---------|-------|------|------|------|
| **TrustMark** | Adobe | MIT | ~500 | 活跃 | 100-bit, ONNX/JS/Rust, C2PA | 不是面向终端用户的 CLI 工具 |
| **Meta Seal** | Meta FAIR | MIT | ~2000 | 活跃 | SOTA 鲁棒性, 多模态 | 重型框架，非轻量工具 |
| **invisible-watermark** | ShieldMnt | MIT | 1852 | **停滞 (>12月)** | SD 推荐，下载量大 | 不够鲁棒，不基于深度学习 |
| **blind-watermark** | 个人 | MIT | 6136 | 停滞 | Stars 多 | DCT/SVD，非深度学习 |
| **SynthID** | Google | 部分开源 | - | 活跃 | 多模态，Google 生态 | 平台级，不可独立使用 |

### 2.2 StegMark 的差异化定位

**用户为什么不用 TrustMark / Meta Seal？**

| 维度 | TrustMark / Meta Seal | **StegMark** |
|------|----------------------|-------------|
| 安装 | 需了解库 API | `pip install stegmark` 零配置 |
| 使用 | 写 Python 代码调用 | **一条命令** `stegmark embed photo.png -m "Alice"` |
| 安装体积 | 需 PyTorch (~700MB+) | **仅 ~25MB**（ONNX Runtime） |
| 批量处理 | 自己写循环 | **内置** `stegmark embed ./photos/ -m "Alice" -r` |
| 中国用户 | 英文文档，HF 被墙 | **中文文档，ModelScope 镜像** |
| 引擎选择 | 单一引擎 | **多引擎切换**（TrustMark/自研） |
| 输出格式 | 需自己解析 | **人类友好 + JSON 结构化** |

**核心差异化：不是"又一个水印库"，而是"水印领域的 ffmpeg"——统一接口、极简 UX、多引擎。**

### 3.3 HiDDeN 架构（自研引擎 v0.2）

#### Encoder 网络

```
输入: cover_image [B, 3, H, W] (归一化 [0,1]) + message [B, L] (二进制)

阶段1 — 图像特征提取 (4 层 ConvBNReLU):
  Conv2d(3→64, k=3, p=1) → BN → ReLU      [B, 64, H, W]
  Conv2d(64→64, k=3, p=1) → BN → ReLU     [B, 64, H, W]
  Conv2d(64→64, k=3, p=1) → BN → ReLU     [B, 64, H, W]
  Conv2d(64→64, k=3, p=1) → BN → ReLU     [B, 64, H, W]

阶段2 — 消息空间扩展与拼接:
  message [B,L] → expand → [B, L, H, W]
  concat(features[B,64], cover[B,3], message[B,L]) → [B, 64+3+L, H, W]

阶段3 — 融合输出:
  Conv2d(64+3+L→64, k=3, p=1) → BN → ReLU
  Conv2d(64→3, k=1, p=0)  # 无 BN/激活，输出残差

输出: encoded = cover + strength * residual  (加性残差嵌入)
参数量: ~180K
```

#### Decoder 网络

```
输入: noised_image [B, 3, H, W]

7 层 ConvBNReLU (64 channels, 感受野 15x15，跨 JPEG 8x8 块边界):
  Conv2d(3→64) → [Conv2d(64→64)] × 6 → BN → ReLU

Global Average Pooling → [B, 64]
Linear(64→L) → [B, L]  (raw logits)

推理: sigmoid > 0.5 → 二值化
参数量: ~227K
```

#### Discriminator

```
3 层 Conv2d(64) + GAP + Linear → [B, 1] (real/fake)
参数量: ~75K

总模型: ~482K 参数，权重文件仅 ~2MB
```

#### 噪声层（训练时）

| 噪声类型 | 实现方式 | 参数 | 权重 |
|---------|---------|------|------|
| Identity | 直通 | - | 10% |
| **JPEG 压缩** | DiffJPEG (STE近似) | QF 50-95 | **35%** |
| 缩放 | F.interpolate 双线性 | scale 0.5-1.0 | 15% |
| 高斯模糊 | 固定核卷积 | sigma 0.5-2.0 | 10% |
| 裁剪 | 随机子区域 | 保留 50%-90% | 10% |
| Dropout | 像素级置零 | p 0.1-0.3 | 10% |
| 高斯噪声 | N(0, σ²) | sigma 0.01-0.05 | 10% |

#### 损失函数

```
L_total = λ_i · L_image + λ_m · L_message + λ_a · L_adversarial

L_image     = MSE(encoded, cover) + 0.5 · LPIPS(encoded, cover)
L_message   = BCEWithLogits(decoded, original_message)
L_adversarial = BCE(D(encoded), 1)  # 欺骗判别器

权重调度:
  Epoch 1-50:   λ_m=5.0,  λ_a=0      (先学嵌入)
  Epoch 51-100: λ_m=8.0,  λ_a=0.0005 (加入对抗)
  Epoch 101+:   λ_m=10.0, λ_a=0.001  (全噪声强化)
```

### 3.4 StegaStamp 架构（自研引擎 v0.4）

相比 HiDDeN 的关键升级：

| 维度 | HiDDeN | StegaStamp-lite |
|------|--------|----------------|
| Encoder | 平坦 ConvBNReLU | **U-Net** 多尺度编码 |
| Decoder | CNN + GAP | **STN 几何校正** + CNN |
| 噪声层 | 数字攻击 | + 透视变换、颜色偏移、运动模糊 |
| 容量 | 48 bits | **100 bits** |
| 参数量 | ~482K | **~15M** |

### 3.5 图像预处理/后处理流水线

**嵌入流程**:
```
原图 [H_orig, W_orig] → 双三次缩放到 128/256 → Encoder → 残差
→ 残差双三次上采样到 [H_orig, W_orig] → 原图 + 残差 → clamp → 保存
```

**提取流程**:
```
水印图 [H, W] → 双三次缩放到 128/256 → Decoder → logits
→ sigmoid > 0.5 → BCH 纠错 → 消息
```

**格式处理**:
- RGBA → 白底 RGB 转换
- 保留 EXIF/ICC Profile 元数据
- 默认输出 PNG（避免 JPEG 二次压缩损失）

### 3.6 水印编解码方案

**编码流程**: 文本 → UTF-8 → 二进制比特串 → BCH 编码 → 网络输入

**BCH 纠错码配置**:

| 版本 | 配置 | 信息位 | 校验位 | 纠错能力 |
|------|------|--------|--------|---------|
| v0.1 | TrustMark 内置 | 100 | - | 取决于 TrustMark |
| v0.2 | BCH(63, 36, t=5) | 36 bits | 27 bits | 纠 5 bit |
| v0.3 | BCH(127, 64, t=10) | 64 bits | 63 bits | 纠 10 bit |

**消息格式协议**:
```
[2-bit 类型头] [N-bit 载荷]
  00 = 原始二进制
  01 = UTF-8 文本
  10 = UUID/哈希
  11 = 保留
```

## 4. CLI 与 API 设计

### 4.1 CLI 命令总览

```
stegmark <command> [options] [arguments]

Commands:
  embed      将水印嵌入图片
  extract    从图片中提取水印
  verify     验证图片是否包含指定水印
  info       显示图片的水印元信息
  benchmark  评估水印鲁棒性
  config     管理配置项
```

### 4.2 核心命令详细定义

#### `stegmark embed`

```
stegmark embed [OPTIONS] <INPUT> -m <MESSAGE> [-o <OUTPUT>]
```

| 长名 | 短名 | 类型 | 默认值 | 描述 |
|------|------|------|--------|------|
| `--message` | `-m` | str | — | 水印文本（与 `--bits` 互斥） |
| `--bits` | `-b` | str | — | 十六进制二进制数据 |
| `--output` | `-o` | path | `{name}_wm.{ext}` | 输出路径 |
| `--strength` | `-s` | float | `1.0` | 强度 0.1-3.0 |
| `--format` | `-f` | str | 同输入 | 输出格式 png/jpeg/webp |
| `--quality` | `-Q` | int | `95` | JPEG/WebP 质量 |
| `--engine` | `-e` | str | `auto` | 引擎: auto/trustmark/hidden/stegastamp |
| `--overwrite` | `-y` | flag | false | 覆盖已有文件 |
| `--recursive` | `-r` | flag | false | 递归处理子目录 |
| `--workers` | `-w` | int | `4` | 批量并行数 |
| `--compare` | — | flag | false | 生成对比报告 |
| `--json` | `-J` | flag | false | JSON 输出 |

#### `stegmark extract`

```
stegmark extract [OPTIONS] <INPUT>
```

| 长名 | 短名 | 类型 | 默认值 | 描述 |
|------|------|------|--------|------|
| `--mode` | — | str | `text` | text/bits |
| `--engine` | `-e` | str | `auto` | 引擎选择 |
| `--recursive` | `-r` | flag | false | 递归子目录 |

#### `stegmark verify`

退出码语义化：`0` 匹配 / `1` 不匹配 / `2` 错误。可嵌入 CI/CD。

#### `stegmark benchmark`

内置 12 种攻击测试，输出鲁棒性报告（table/csv/json/html）。

### 4.3 输出格式

**人类友好（默认）**:
```
$ stegmark embed photo.png -m "Alice 2026"

StegMark v0.1.0
───────────────────────────────────
  Input   : photo.png (1920x1080, PNG)
  Message : Alice 2026 (80 bits)
  Engine  : trustmark
───────────────────────────────────
  Embedding... ━━━━━━━━━━━━━━ 100%
───────────────────────────────────
  Output  : photo_wm.png
  PSNR    : 42.3 dB (excellent)

```

**安静模式** (`-q`): 仅输出路径或文本
**JSON 模式** (`--json`): 结构化输出，统一信封格式

### 4.4 Python API

#### 简单函数层

```python
import stegmark

# 嵌入
result = stegmark.embed("photo.png", "Alice 2026")
result.save("photo_wm.png")
print(f"PSNR: {result.psnr:.1f} dB")

# 提取
extracted = stegmark.extract("photo_wm.png")
print(extracted.message)  # "Alice 2026"

# 验证
if stegmark.verify("photo_wm.png", "Alice 2026"):
    print("Watermark verified!")
```

#### 高级类层

```python
from stegmark import StegMark

with StegMark(device="cuda", engine="hidden", strength=1.2) as sm:
    for img_path in Path("./originals").glob("*.png"):
        result = sm.embed(img_path, message="Studio X 2026")
        result.save(Path("./protected") / img_path.name)
```

#### 返回值设计

- `EmbedResult`: image (ndarray), psnr, ssim, elapsed, save(), to_pil(), compare()
- `ExtractResult`: found (bool), confidence, message, bits, 支持 `if result:` 判断
- `VerifyResult`: matched (bool), confidence, bit_accuracy
- `BenchmarkResult`: attack_results (dict), to_table(), to_dataframe()

所有 Result 为 frozen dataclass，支持布尔判断。

#### 异常层次

```
StegMarkError
 ├── InvalidInputError
 │    ├── MessageTooLongError
 │    ├── InvalidStrengthError
 │    ├── UnsupportedFormatError
 │    └── ImageTooSmallError
 ├── FileError (ImageReadError, ImageWriteError)
 ├── ModelError (ModelNotFoundError, ModelDownloadError)
 ├── WatermarkError (WatermarkNotFoundError)
 └── DeviceError (CUDANotAvailableError)
```

每个异常带 `hint` 字段，提供可操作建议。

### 4.5 配置系统

**文件**: `~/.stegmark/config.toml`
**优先级**: 命令行参数 > 环境变量 (STEGMARK_*) > 配置文件 > 默认值

关键配置项: device, strength, engine, model_dir, auto_download, workers, threshold

### 5.1 硬件约束

| 参数 | 值 |
|------|-----|
| GPU | RTX 4060 Ti 8GB GDDR6 |
| CUDA Cores | 4352 |
| FP32 | 22.1 TFLOPS |

### 5.2 训练数据

| 数据集 | 图片数 | 用途 | License |
|--------|--------|------|---------|
| COCO 2017 train (子集) | 10K-118K | 主训练集 | CC BY 4.0 ✓ |
| COCO 2017 val | 5K | 验证集 | CC BY 4.0 ✓ |
| Unsplash Dataset | 按需 | 补充训练 | Unsplash License ✓ |

**注意**: 不使用 DIV2K（仅限学术使用，商业发布有法律风险）。

### 5.3 HiDDeN 训练配置

```
图像尺寸:   128×128 (v0.2), 256×256 (v0.3)
Batch Size: 32 (128×128) / 16 (256×256)
消息长度:   48 bits → BCH 编码后 63 bits
优化器:     Adam (lr=1e-3, weight_decay=1e-5)
调度器:     CosineAnnealingWarmRestarts (T_0=50, T_mult=2)
混合精度:   128×128 不需要 / 256×256 推荐 AMP
显存占用:   128×128 ~2.5GB / 256×256 ~5.5GB
训练时间:   128×128 ~2小时 / 256×256 ~8小时
```

### 5.4 课程学习策略

| 阶段 | Epoch | 噪声 | 目标 |
|------|-------|------|------|
| Phase 1: 基础嵌入 | 1-50 | 无噪声 | BER < 0.001 |
| Phase 2: 空域攻击 | 51-100 | Dropout + Crop + Gaussian | BER < 0.05 (有攻击) |
| Phase 3: JPEG 攻击 | 101-200 | JPEG QF=70-90 为主 | JPEG Q75 BER < 0.05 |
| Phase 4: 全噪声强化 | 201-300 | 全种类，强度增大 | JPEG Q50 BER < 0.10 |

### 5.5 收敛策略

- 早停: 监控验证集 BER，patience = 30 epochs
- 检查点: 每 10 epoch 保存，保留最佳 BER 模型
- 失败处理: BER 不下降 → 降噪声强度；PSNR < 35dB → 增大 λ_i

## 9. 风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| TrustMark/Meta Seal 已存在 | **高** | 定位为上层工具而非竞争，混合策略 |
| PyTorch 依赖太重 | **高** | 推理用 ONNX Runtime，PyTorch 仅训练时需要 |
| HuggingFace 中国被墙 | **中** | ModelScope 镜像 + 手动下载选项 |
| 可微 JPEG 实现复杂 | **中** | 参考 DiffJPEG + MBRS 方案，已有成熟开源实现 |
| 单人项目维护压力 | **中** | 先做好核心功能，不追求大而全；善用 CI 自动化 |
| 训练数据版权 | **低** | 不用 DIV2K，只用 COCO (CC BY 4.0) + Unsplash |
| 抗截图不切实际 | **低** | 标记为 experimental，降低用户预期 |
| 水印被恶意使用 | **低** | README 加使用声明，不提供 removal 模块 |
